# End-to-End Validation (`Detector.IntegrationTests`)

The `Detector.IntegrationTests` project contains elevated integration tests that validate the full detection pipeline against real ETW events — not mocks, not injected events, but actual kernel file-I/O telemetry generated by the simulator on the test machine.

## Requirement: Administrator Elevation

ETW kernel file-I/O sessions require Administrator privileges. All tests in this project check elevation at the start and **skip automatically** if the process is not elevated:

```csharp
if (!ElevationHelper.IsElevated())
    return; // or Assert.Skip
```

To run integration tests, execute from an elevated PowerShell prompt:
```powershell
dotnet test tests\Detector.IntegrationTests
```

---

## Architecture: Headless Pipeline

The tests spin up a bare pipeline using `Host.CreateDefaultBuilder` — the same DI/service graph used by `Detector.App`, but without the WPF GUI or tray:

```csharp
var host = Host.CreateDefaultBuilder()
    .ConfigureServices(services =>
    {
        services.AddOptions<ActDefendOptions>()...
        services.AddCollector()
                .AddFeatures()
                .AddDetection()
                .AddEntropy()
                .AddStorage();
        services.AddHostedService<PipelineHostService>();
    })
    .Build();
```

This ensures the test exercises the exact same code paths as production, with no test-specific stubs in the hot path.

## Test Scenarios

### Ransomware Detection (True Positive)

1. Start the headless pipeline host.
2. Run `SimulatorRunner.RunRansomwareWorkload()` against a dedicated `test-workspace` directory.
3. Wait for the expected number of orchestration ticks to elapse.
4. Query `IAlertRepository.GetAllAsync()` and assert ≥ 1 confirmed alert exists in SQLite.

The goal is to verify that:
- ETW events generated by the simulator are ingested by `EtwEventCollector`.
- `FeatureExtractor` computes burst metrics that exceed Stage 1 thresholds.
- `EntropySamplingEngine` finds the `.locked` renamed files via extension probing and confirms high entropy.
- `AlertRepository` persists the resulting `DetectionAlert` to the SQLite database.

### Benign Non-Detection (True Negative)

1. Start the headless pipeline host.
2. Run `SimulatorRunner.RunBenignWorkload()` against a `test-workspace` directory.
3. Wait for the detection window to elapse.
4. Assert that `IAlertRepository.GetAllAsync()` returns 0 alerts.

The goal is to verify no false alerts are raised from the low-rate, low-entropy benign file writes.

---

## Test Database Isolation

Each integration test run uses a temporary SQLite file created for that test (not the production `actdefend.db`). The database path is injected via configuration overrides on the test host builder. The test database is deleted after the test completes.

---

## Relationship to Unit Tests

| Aspect | Unit Tests | Integration Tests |
|---|---|---|
| ETW | No (mocked events) | Yes (real kernel session) |
| Elevation | Not required | Required |
| Speed | Fast (ms) | Slow (seconds — waits for ETW ticks) |
| Coverage | Scoring math, feature logic, entropy calc | Full pipeline from ETW to SQLite |
| Location | `tests/Detector.UnitTests` | `tests/Detector.IntegrationTests` |

---

## Known Limitations

- Integration tests are inherently environment-dependent. Other processes running heavy file I/O during the test can affect event counts observed by the test pipeline.
- The test workspace must be on a local NTFS volume. Network paths may not generate ETW FileIO events from the kernel provider.
- Tests run serially (not in parallel) to avoid multiple simultaneous ETW sessions from conflicting.
